# Dogfood 实验 · 双项目并发（v0.3 多项目能力实战检验）

**日期**：2026-07-19 · **平台版本**：v0.4（main `b98ee11`）· **实验人**：平台作者（全程无人干预,仅观察）

## 实验设置

- **命题**：多项目并发（v0.3 第 4 项）不只要在 E2E 里跑通,要在**生产配置**下同时交付两个真实需求
- **两个需求**（都是平台自己要用的纯内核,可移植回平台）：
  - **A（项目 #13）成本日报内核**：usage 记录 → Markdown 成本报告（按日/agent/模型 + 占比 + 环比）——候选并入汇报器
  - **B（项目 #14）教训检索内核**：中英双语 bigram/tf-idf 检索教训——直击团队记忆"朴素关键词匹配"这个已知弱点
- **配置**：生产默认。协调者/架构师/BA = opus,工兵 = sonnet,书记官 = haiku;全部质检门开启（自测/审查/QA/质疑/终审/**集成回归门首次生产运行**）;`approval_policy=budget_only`;`max_concurrent_projects=2`;预算 $30×2
- **创建方式**：两个 POST 相隔 **114ms**,双双自动起跑

## 并发行为观察（多项目能力的实战证据）

- **两场 kickoff 交错进行**：`meeting.started` 事件 3555/3557 背靠背,收敛裁决 3562/3563 相邻——不是排队,是真并行
- **两场会议都第 1 轮收敛**（质疑者无异议提前散会）——收敛环的提前出口在省钱
- **会话/成本/交付物零串扰**：两仓各自只有自己的文件;usage 按 project_id 精确归账（$14.88 vs $10.56）
- **WS 增量合并首战**：双项目高频事件流下前端不再全量重拉（v0.4 第 8 项的直接受益场景）

## 质检环节的真实工作量

- **设计环抓到真问题**：A 的架构方案被质疑者第 1 轮提出 **3 个 issue**,架构师修订后第 2 轮放行（B 第 1 轮直接放行）
- **终审门抓到真缺口**：A 的核心实现任务 #45 通过了自测/审查/QA/质疑四道门,却被协调者终审拦下（`complete:false, gaps:1`）——返工后第二次终审通过。这正是终审存在的意义:对照验收标准查"缺不缺",而不是重看代码
- **集成回归门生产首秀**：6 次合并后全项目自测 **6/6 全过**（本次没有回归,门在,记录在）

## 交付物实测（不信自述,亲手跑）

- **A**：`node --test` **17/17 通过**;demo 输出完整成本报告——按日/agent/模型三表齐全,占比合计 100%,环比节正确
- **B**：`node --test` **10/10 通过**;demo 检索质量佳:「timeout 或超时」简报同时命中英文 `timeout` 与中文「超时」,tags 加权肉眼可见,`why` 字段给出命中词
